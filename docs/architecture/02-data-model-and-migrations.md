# Data model and migrations

This document describes the desired schema, store interface, ID strategy, and migration approach. Build order is in [roadmap.md](../roadmap.md).

---

## Design decisions

| Question | Decision |
|----------|----------|
| ID generation | **ULID** (`github.com/oklog/ulid/v2`) — time-sortable, lexicographically ordered, URL-safe |
| Store interface structure | **Per-domain sub-interfaces** composed into a root `Store` — better testability |
| Circular reference (accounts ↔ media_attachments) | **Split migrations** — accounts first (no FK), then media_attachments, then ALTER TABLE |
| media_attachments ↔ statuses association | `status_id` column added to media_attachments; FK constraint added in statuses migration |
| Status edit history | **Phase 1** — `status_edits` table included from the start |
| Mutes schema | `hide_notifications BOOLEAN NOT NULL DEFAULT TRUE` — Mastodon-compatible |
| Server filters | **Simple** — `phrase`, `scope` (public_timeline\|all), `action` (warn\|hide) |
| Down migrations | **Destructive** `DROP TABLE … CASCADE` |
| OAuth authorization code lifecycle | Delete on use — no replay possible; no `used` flag needed |
| Transaction support | `WithTx(ctx, fn)` wrapping sqlc-generated `WithTx(tx)` |

## ID Generation

**Choice: ULID via `github.com/oklog/ulid/v2`**

### Rationale

Mastodon clients use `max_id` and `since_id` as pagination cursors — they pass the last-seen status ID as a string bound and expect the server to return only records "older than" that ID. For this to work correctly without a separate `created_at` comparison, IDs must be **lexicographically sortable by creation time**.

| Strategy | Sortable | URL-safe | Length | Verdict |
|----------|----------|----------|--------|---------|
| UUID v4 | ✗ | ✗ (with hyphens) | 36 chars | Pagination requires `ORDER BY created_at`, cursor must carry timestamp |
| nanoid | ✗ | ✓ | ~21 chars | Same problem as UUID — time not encoded |
| **ULID** | **✓** | **✓** | **26 chars** | **Correct cursor pagination by ID alone** |
| Snowflake | ✓ | ✓ | ~19 chars | Requires shared epoch + machine ID coordination |

With ULID, `WHERE id < $maxID ORDER BY id DESC` is correct and efficient. The first 10 Crockford base32 characters encode a 48-bit millisecond timestamp; the remaining 16 characters are random. A **monotonic entropy source** (`ulid.Monotonic`) is used per-request to guarantee uniqueness within the same millisecond.

### Generation helper


All service-layer code calls `uid.New()` before inserting — IDs are never generated by the database.

## 5. `sqlc` configuration

**File:** `sqlc.yaml` (project root)


**Notes:**
- `emit_pointers_for_null_types: true` — nullable `TEXT` columns become `*string` in Go, making nil-checks explicit.
- `emit_interface: true` — generates a `Querier` interface alongside `*Queries`; the `Store` interface below embeds/wraps it.
- `emit_prepared_queries: false` — avoids prepared statement overhead on pgBouncer-proxied pools (relevant for Phase 3).
- `sql_package: "pgx/v5"` — emits `pgx.Rows`, `pgx.Row` etc. rather than `database/sql` types.

## 7. `Store` Interface

**File:** `internal/store/store.go`

### Design choice: per-domain sub-interfaces

Per-domain sub-interfaces are composed into a root `Store`. Rationale:

- **Testability** — a service that only reads accounts can accept an `AccountStore` in its constructor; tests mock only that interface rather than the entire store.
- **Modularity** — if a service sub-package is extracted to its own binary in Phase 3, it takes only the sub-interface it needs.
- **Discoverability** — it's immediately clear which database operations a given service uses.

The downside (more interfaces to maintain) is minor compared to the benefits at this scale.
