# ADR 02 — Data Model & Database Layer

> **Status:** Draft v0.1 — Feb 24, 2026

## Design Decisions (answered before authoring)

| Question | Decision |
|----------|----------|
| ID generation | **ULID** (`github.com/oklog/ulid/v2`) — time-sortable, lexicographically ordered, URL-safe |
| Store interface structure | **Per-domain sub-interfaces** composed into a root `Store` — better testability |
| Circular reference (accounts ↔ media_attachments) | **Split migrations** — accounts first (no FK), then media_attachments, then ALTER TABLE |
| media_attachments ↔ statuses association | `status_id` column added to media_attachments; FK constraint added in statuses migration |
| Status edit history | **Phase 1** — `status_edits` table included from the start |
| Mutes schema | `hide_notifications BOOLEAN NOT NULL DEFAULT TRUE` — Mastodon-compatible |
| Server filters | **Simple** — `phrase`, `scope` (public_timeline\|all), `action` (warn\|hide) |
| Down migrations | **Destructive** `DROP TABLE … CASCADE` |
| OAuth authorization code lifecycle | Delete on use — no replay possible; no `used` flag needed |
| Transaction support | `WithTx(ctx, fn)` wrapping sqlc-generated `WithTx(tx)` |

## 1. ID Generation

**Choice: ULID via `github.com/oklog/ulid/v2`**

### Rationale

Mastodon clients use `max_id` and `since_id` as pagination cursors — they pass the last-seen status ID as a string bound and expect the server to return only records "older than" that ID. For this to work correctly without a separate `created_at` comparison, IDs must be **lexicographically sortable by creation time**.

| Strategy | Sortable | URL-safe | Length | Verdict |
|----------|----------|----------|--------|---------|
| UUID v4 | ✗ | ✗ (with hyphens) | 36 chars | Pagination requires `ORDER BY created_at`, cursor must carry timestamp |
| nanoid | ✗ | ✓ | ~21 chars | Same problem as UUID — time not encoded |
| **ULID** | **✓** | **✓** | **26 chars** | **Correct cursor pagination by ID alone** |
| Snowflake | ✓ | ✓ | ~19 chars | Requires shared epoch + machine ID coordination |

With ULID, `WHERE id < $maxID ORDER BY id DESC` is correct and efficient. The first 10 Crockford base32 characters encode a 48-bit millisecond timestamp; the remaining 16 characters are random. A **monotonic entropy source** (`ulid.Monotonic`) is used per-request to guarantee uniqueness within the same millisecond.

### Generation helper

```go
// internal/uid/uid.go
package uid

import (
    "math/rand"
    "sync"
    "time"

    "github.com/oklog/ulid/v2"
)

var (
    entropy = ulid.Monotonic(rand.New(rand.NewSource(time.Now().UnixNano())), 0)
    mu      sync.Mutex
)

// New returns a new, time-sortable ULID string.
func New() string {
    mu.Lock()
    defer mu.Unlock()
    return ulid.MustNew(ulid.Timestamp(time.Now()), entropy).String()
}
```

All service-layer code calls `uid.New()` before inserting — IDs are never generated by the database.

## 2. Circular Reference Resolution

**Problem:** `accounts.avatar_media_id` and `accounts.header_media_id` reference `media_attachments(id)`, while `media_attachments.account_id` references `accounts(id)`. Neither table can be created with both FKs if the other doesn't yet exist.

**Solution: three-step split migration**

1. **Migration 000001** — Create `accounts` *without* `avatar_media_id` / `header_media_id` columns.
2. **Migration 000002** — Create `media_attachments` with `account_id REFERENCES accounts(id)`. Also add `status_id TEXT` column (no FK yet — added in migration 000005).
3. **Migration 000003** — `ALTER TABLE accounts ADD COLUMN avatar_media_id … ADD COLUMN header_media_id …` with FK references to `media_attachments`.

The `media_attachments ↔ statuses` cycle is resolved the same way: `media_attachments.status_id` FK constraint is added at the end of migration 000005 after `statuses` is created.

This avoids `DEFERRABLE INITIALLY DEFERRED` constraints, which leak transaction semantics into the application layer and are easy to misuse.

```sql
CREATE TABLE accounts (
    id            TEXT PRIMARY KEY,
    username      TEXT NOT NULL,
    domain        TEXT,                         -- NULL for local accounts
    display_name  TEXT,
    note          TEXT,                         -- bio (HTML)
    -- avatar_media_id and header_media_id added in migration 000003

    updated_at    TIMESTAMPTZ NOT NULL DEFAULT NOW(),
```

### `000005_create_statuses.up.sql`

```sql
CREATE TABLE statuses (
    id               TEXT PRIMARY KEY,
    uri              TEXT NOT NULL UNIQUE,       -- AP IRI
    account_id       TEXT NOT NULL REFERENCES accounts(id),
    text             TEXT,                       -- original Markdown/plain text
    content          TEXT,                       -- rendered HTML
    content_warning  TEXT,
    visibility       TEXT NOT NULL,              -- 'public'|'unlisted'|'private'|'direct'
    language         TEXT,
    in_reply_to_id   TEXT REFERENCES statuses(id),
    reblog_of_id     TEXT REFERENCES statuses(id),
    ap_id            TEXT NOT NULL UNIQUE,
    ap_raw           JSONB,
    sensitive        BOOLEAN NOT NULL DEFAULT FALSE,
    local            BOOLEAN NOT NULL DEFAULT TRUE,
    edited_at        TIMESTAMPTZ,               -- NULL if never edited
    replies_count    INT NOT NULL DEFAULT 0,
    reblogs_count    INT NOT NULL DEFAULT 0,
    favourites_count INT NOT NULL DEFAULT 0,
    created_at       TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at       TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    deleted_at       TIMESTAMPTZ                -- soft delete
);

-- Account timeline: paginated by id DESC.
CREATE INDEX idx_statuses_account ON statuses (account_id, id DESC)
    WHERE deleted_at IS NULL;

-- Local public timeline (used constantly; partial index keeps it tight).
CREATE INDEX idx_statuses_local_public ON statuses (id DESC)
    WHERE local = TRUE AND visibility = 'public' AND deleted_at IS NULL;

-- Federated public timeline (adds remote public posts).
CREATE INDEX idx_statuses_public ON statuses (id DESC)
    WHERE visibility = 'public' AND deleted_at IS NULL;

-- Thread context: find all replies to a given status.
CREATE INDEX idx_statuses_reply ON statuses (in_reply_to_id)
    WHERE in_reply_to_id IS NOT NULL AND deleted_at IS NULL;

-- Finding boosts of a given status (for unboost and reblogged_by queries).
CREATE INDEX idx_statuses_reblog ON statuses (reblog_of_id, account_id)
    WHERE reblog_of_id IS NOT NULL;

-- Now that statuses exists, add the FK constraint on media_attachments.status_id.
ALTER TABLE media_attachments
    ADD CONSTRAINT fk_media_status
    FOREIGN KEY (status_id) REFERENCES statuses(id) ON DELETE SET NULL;
```

### `000005_create_statuses.down.sql`

```sql
ALTER TABLE media_attachments DROP CONSTRAINT IF EXISTS fk_media_status;
DROP TABLE IF EXISTS statuses CASCADE;
```

## 5. `sqlc` Configuration

**File:** `sqlc.yaml` (project root)

```yaml
version: "2"
sql:
  - engine: "postgresql"
    queries: "internal/store/postgres/queries"
    schema:  "internal/store/migrations"
    gen:
      go:
        package:                     "db"
        out:                         "internal/store/postgres/generated"
        sql_package:                 "pgx/v5"
        emit_interface:              true
        emit_json_tags:              true
        emit_db_tags:                false
        emit_prepared_queries:       false
        emit_exact_table_names:      false
        emit_empty_slices:           true
        emit_pointers_for_null_types: true
        overrides:
          # TIMESTAMPTZ → time.Time (pgx/v5 handles this natively; explicit for clarity)
          - db_type: "pg_catalog.timestamptz"
            go_type:
              import: "time"
              type:   "Time"
          # TEXT[] → []string
          - db_type: "text[]"
            go_type:
              type: "[]string"
          # JSONB → json.RawMessage (preserves raw bytes; services unmarshal what they need)
          - db_type: "jsonb"
            go_type:
              import: "encoding/json"
              type:   "RawMessage"
```

**Notes:**
- `emit_pointers_for_null_types: true` — nullable `TEXT` columns become `*string` in Go, making nil-checks explicit.
- `emit_interface: true` — generates a `Querier` interface alongside `*Queries`; the `Store` interface below embeds/wraps it.
- `emit_prepared_queries: false` — avoids prepared statement overhead on pgBouncer-proxied pools (relevant for Phase 3).
- `sql_package: "pgx/v5"` — emits `pgx.Rows`, `pgx.Row` etc. rather than `database/sql` types.

```sql
      UNION ALL

WHERE deleted_at IS NULL
  AND visibility = 'public'

ORDER BY id DESC
LIMIT $3;

WHERE account_id = $1
  AND deleted_at IS NULL

ORDER BY id DESC
LIMIT $3;

        SELECT in_reply_to_id FROM statuses WHERE id = $1 AND deleted_at IS NULL
```

## 7. `Store` Interface

**File:** `internal/store/store.go`

### Design choice: per-domain sub-interfaces

Per-domain sub-interfaces are composed into a root `Store`. Rationale:

- **Testability** — a service that only reads accounts can accept an `AccountStore` in its constructor; tests mock only that interface rather than the entire store.
- **Modularity** — if a service sub-package is extracted to its own binary in Phase 3, it takes only the sub-interface it needs.
- **Discoverability** — it's immediately clear which database operations a given service uses.

The downside (more interfaces to maintain) is minor compared to the benefits at this scale.

## 8. `db.go` — pgx Pool Setup

**File:** `internal/store/postgres/db.go`

```go
package postgres

import (
    "context"
    "fmt"
    "time"

    "github.com/jackc/pgx/v5/pgxpool"

    "github.com/yourorg/monstera/internal/config"
    "github.com/yourorg/monstera/internal/store"
    "github.com/yourorg/monstera/internal/store/postgres/generated"
)

// NewPool opens and validates a pgx connection pool from the application Config.
//
// Connection pool sizing rationale:
//   - MaxConns: matches config.DatabaseMaxOpenConns (default 20).
//     Each replica pod opens up to 20 connections. With 3 replicas that is
//     60 server-side connections — well within PostgreSQL's default max_connections
//     of 100. Adjust upward only after profiling; more connections means more
//     memory pressure on PostgreSQL and contention on locks.
//   - MinConns: 2 — keep a small warm pool so the first requests after a cold
//     start do not incur handshake latency.
//   - MaxConnLifetime: 1h — rotates connections regularly to avoid stale
//     server-side state and to spread reconnects over time.
//   - MaxConnIdleTime: 5m — releases idle connections back to the OS promptly,
//     important in autoscale environments where pods may sit idle for long periods.
//   - HealthCheckPeriod: 30s — pgxpool pings idle connections; avoids
//     "connection reset by peer" errors after firewall idle-timeout.
func NewPool(ctx context.Context, cfg *config.Config) (*pgxpool.Pool, error) {
    pcfg, err := pgxpool.ParseConfig(cfg.DatabaseURL)
    if err != nil {
        return nil, fmt.Errorf("parse database URL: %w", err)
    }

    pcfg.MaxConns            = int32(cfg.DatabaseMaxOpenConns)
    pcfg.MinConns            = 2
    pcfg.MaxConnLifetime     = time.Hour
    pcfg.MaxConnIdleTime     = 5 * time.Minute
    pcfg.HealthCheckPeriod   = 30 * time.Second

    pool, err := pgxpool.NewWithConfig(ctx, pcfg)
    if err != nil {
        return nil, fmt.Errorf("open pool: %w", err)
    }

    if err := pool.Ping(ctx); err != nil {
        pool.Close()
        return nil, fmt.Errorf("ping database: %w", err)
    }

    return pool, nil
}

// PostgresStore wraps the sqlc-generated *db.Queries and satisfies store.Store.
// It holds the pool so that WithTx can begin new transactions.
type PostgresStore struct {
    pool    *pgxpool.Pool
    queries *db.Queries
}

// New constructs a PostgresStore from an open pool.
func New(pool *pgxpool.Pool) store.Store {
    return &PostgresStore{
        pool:    pool,
        queries: db.New(pool),
    }
}
```

## 9. Transaction Helper — `WithTx`

```go
// WithTx begins a transaction, wraps the connection in a new *db.Queries, and
// calls fn with a transaction-scoped PostgresStore.
//
// If fn returns a non-nil error the transaction is rolled back and that error
// is returned to the caller. If fn returns nil the transaction is committed.
//
// The deferred Rollback is a no-op after a successful Commit — pgx guarantees
// this, so there is no double-rollback risk.
//
// Nested WithTx calls are NOT supported; fn must not call WithTx on the store
// it receives. PostgreSQL savepoints are not used — if savepoint semantics are
// needed they must be implemented explicitly with pgx.Tx.
func (s *PostgresStore) WithTx(ctx context.Context, fn func(store.Store) error) error {
    tx, err := s.pool.Begin(ctx)
    if err != nil {
        return fmt.Errorf("begin transaction: %w", err)
    }
    defer tx.Rollback(ctx) // no-op after Commit

    txStore := &PostgresStore{
        pool:    s.pool,
        queries: s.queries.WithTx(tx),
    }

    if err := fn(txStore); err != nil {
        return err
    }

    return tx.Commit(ctx)
}
```

**Usage pattern in service layer:**

```go
err = store.WithTx(ctx, func(tx store.Store) error {
    if _, err := tx.CreateStatus(ctx, statusParams); err != nil {
        return err
    }
    if err := tx.IncrementRepliesCount(ctx, parentID); err != nil {
        return err
    }
    if _, err := tx.CreateNotification(ctx, notifParams); err != nil {
        return err
    }
    return nil
})
```
